# Второй мозг на квитанциях — полное руководство

> English version: [`brain-notary-guide.md`](brain-notary-guide.md).

**Obsidian + Claude Code + нотариальный слой TimeLayer**

База знаний, которая накапливается, а не переоткрывается; вики, которую пишет не ваша рука; и механизм, который делает проверку корректности **обязательной, неподделываемой, именной и саморазрушающейся при любой правке**.

> Документ сводит воедино три вещи: методологию **LLM Knowledge Base Андрея Карпатого** (паттерн `raw/wiki`, цикл Ingest/Query/Lint — [gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)), проводку из гайда «Второй мозг на Claude + Obsidian» (плагин, MCP, расписание) и нотариальный слой корректности на TimeLayer. Кто такой Карпатый: соосновал OpenAI, руководил ИИ автопилота Tesla, PhD из Стэнфорда, известный преподаватель ИИ. Паттерн — его; нотариальная надстройка закрывает место, которое он сам отмечает как слабое (см. §0 и §18).

## 0. Сначала — честная модель. Без неё механизм соврёт вам

Всё держится на одном различии. Спутаете его — построите систему, которая выглядит надёжной и тихо врёт.

**Нотариальность не проверяет истину и не может.** Квитанция (receipt) доказывает, что *нечто произошло над вот этими байтами в такой-то момент и не менялось потом*. О том, *верно ли содержимое*, она молчит. Заверенная галлюцинация — всё ещё галлюцинация, просто теперь с печатью.

Поэтому мы заверяем не «текст», а **суждение о тексте**: связку из байтов заметки, конкретного источника и вердикта проверки. Квитанция превращается из «вот текст» в «**вот доказательство, что проверка прошла ровно над этими словами, и вот кто за неё отвечает**». Это потолок крипты — и его хватает, если правильно выбрать, *что* заверять и *кто* проверяет.

| Слой | Вопрос | Чем обеспечивается |
|---|---|---|
| **Доступ** | Может ли агент вообще изменить файл | **Ключи** (read-only, узкая область). Не промпты. |
| **Целостность и процесс** | Если изменил — остался ли неподделываемый именной след; прошла ли проверка | **Квитанции** TimeLayer |
| **Корректность** | Верно ли написанное | **Проверяющий** (механика/модель/человек). Ни ключи, ни квитанции этого не дают |

«Неизменяемое», которое нужно в вики, — не «байты нельзя поменять», а **tamper-evident**: правка обнаруживается мгновенно и снимает статус «проверено». Неизменяемость *доверия*, а не байтов. Запретить запись физически умеет только ключ. Держите эту таблицу перед глазами весь оставшийся гайд.

## ЧАСТЬ I. ЗАЧЕМ ЭТО И КАКОЙ ОНО ФОРМЫ (методология Карпатого)

### 1. Почему обычные базы знаний не работают

Большинство работает с документами через LLM так: загрузил файлы, модель ищет релевантные куски на каждый запрос, генерирует ответ. Это **RAG** — и у него четыре врождённые дыры:

- **Нет накопления** — каждый вопрос с нуля.
- **Нет синтеза** — модель не связывает информацию между источниками.
- **Нет обновления** — новые данные не правят старые выводы.
- **Нет структуры** — знание размазано по чанкам без перекрёстных ссылок.

Картина по Карпатому: собрал 50 статей, спросил — модель нашла 3 релевантных чанка из 50. Завтра похожий вопрос — другие 3. Полной картины нет никогда; знание не накапливается, оно переизобретается.

### 2. Идея: персистентная компаундируемая вики

Вместо поиска по сырым документам LLM **инкрементально строит и поддерживает вики** — структурированную коллекцию markdown-файлов с перекрёстными ссылками. Ссылки уже проставлены, противоречия уже отмечены, синтез уже отражает всё прочитанное. Вики становится богаче с каждым источником и каждым вопросом.

Почему именно LLM, а не вы: скучная часть вики — не чтение и не мышление, а **бухгалтерия** (ссылки, обновления, консистентность). Люди бросают вики, потому что затраты на поддержание растут быстрее ценности. LLM не устаёт и обновляет 15 файлов за один проход.

**Разделение ролей:**

| Вы (человек) | LLM (агент) |
|---|---|
| Курируете источники | Суммаризация и индексирование |
| Задаёте направление | Перекрёстные ссылки |
| Задаёте вопросы | Обновление при новых данных |
| Думаете над смыслом | Бухгалтерия и консистентность |

Куда это годится: исследования, внутренняя вики компании (из Slack/митингов/звонков), личное развитие, чтение книг, конкурентный анализ.

### 3. Архитектура: четыре слоя + схема

Карпатый держит три слоя; нотариальный слой добавляет четвёртый (`receipts/`) и карантин (`unverified/`).

| Слой | Назначение | Пишет | Читает | Правило |
|---|---|---|---|---|
| `raw/` | Сырые источники | Вы | LLM | **Никогда не изменять** |
| `wiki/` | Вики от LLM | LLM | Вы | LLM владеет полностью |
| `outputs/` | Отчёты, ответы | LLM | Вы | Можно вернуть в wiki |
| `receipts/` | Квитанции (провенанс + вердикты) | скрипт | скрипт/вы | Append-only |
| `unverified/` | Карантин непроверенного | скрипт | вы | Временный |
| `CLAUDE.md` | Схема (конфиг) | Вы + LLM | LLM | Эволюционирует |

**Структура папок:**

```
my-knowledge-base/
├── raw/                 <- Неизменяемые источники. LLM читает, НИКОГДА не правит
│   ├── articles/        <- статьи, блог-посты
│   ├── papers/          <- исследования
│   ├── notes/           <- личные заметки
│   ├── telegram/        <- посты из Telegram
│   └── images/          <- диаграммы, скриншоты
├── wiki/                <- Вики от LLM. Вы — не трогаете руками
│   ├── index.md         <- каталог всех страниц (LLM читает первым)
│   ├── log.md           <- append-only лог операций (зачаток аудит-следа)
│   └── _templates/      <- шаблоны страниц
├── outputs/             <- отчёты, аналитика
├── receipts/            <- .tlcert/.tlbundle: квитанции источников и вердиктов
│   ├── raw/
│   └── wiki/
├── unverified/          <- заметки без валидной квитанции проверки
└── CLAUDE.md            <- схема для LLM
```

Ключевые файлы: **`wiki/index.md`** — каталог по доменам, LLM читает его первым, чтобы найти релевантные страницы. **`wiki/log.md`** — append-only лог операций (`## [YYYY-MM-DD] операция | описание`); именно сюда мы будем класть id квитанций, а сам лог можно нотарировать. **`CLAUDE.md`** — схема: без неё LLM просто чатбот, с ней — дисциплинированный wiki-maintainer.

## ЧАСТЬ II. СБОРКА БАЗЫ (проводка)

### 4. Установите Claude Code

Скачайте десктоп-приложение Claude с `claude.com/download`, откройте вкладку **Code** — версия, которая читает и пишет файлы на компьютере. Нужен платный план (Claude Pro ~$20/мес; на бесплатном не запустится). Единственная обязательная статья расходов в базовой части. Паттерн не привязан к Claude — Cursor, Codex, любой агент с доступом к файлам тоже подходят.

### 5. Создайте структуру

Скачайте Obsidian с `obsidian.md`, создайте волт. В терминале волта:

```bash
mkdir -p raw/articles raw/papers raw/notes raw/telegram raw/images
mkdir -p wiki/_templates outputs receipts/raw receipts/wiki unverified
```

Или просто скажите Claude Code: «Создай структуру LLM Knowledge Base по паттерну Карпатого, плюс папки `receipts/` и `unverified/`».

### 6. Откройте дверь в хранилище (плагин Local REST API)

В Obsidian: **Settings → Community plugins → включить → Browse → установить Local REST API**. Включите, откройте настройки, скопируйте **API key**. Плагин поднимает локальный сервер на `127.0.0.1:27124`.

**Не закрывайте Obsidian** — дверь работает только пока приложение открыто.

> Это «тот самый плагин». TimeLayer плагином не является — подключается отдельно (§13).

### 7. Подключите Claude через MCP

На вкладке Code, подставив ключ (слово `Bearer` из плагина выбросьте, оставьте строку после него):

```bash
claude mcp add-json obsidian-vault '{
  "type": "stdio",
  "command": "uvx",
  "args": ["mcp-obsidian"],
  "env": {
    "OBSIDIAN_API_KEY": "ВСТАВЬТЕ-СВОЙ-КЛЮЧ",
    "OBSIDIAN_HOST": "127.0.0.1",
    "OBSIDIAN_PORT": "27124"
  }
}'
```

`mcp-obsidian` — проверенный мост с тысячами звёзд на GitHub. Проверка: «выведи список всех файлов в моём хранилище».

### 8. CLAUDE.md — мозг системы

Самый важный файл. Содержит шесть блоков:

1. **Архитектура** — описание четырёх слоёв (раздел 3).
2. **Frontmatter-схема** — метаданные каждой вики-страницы. Базовая схема Карпатого, расширенная полями нотариального слоя:

```yaml
---
type: entity | concept | project | person | summary
domain: ai | infra | content | personal
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: ["[[raw/articles/file]]@a1b2c3"]   # источник + ХЕШ версии (наше добавление)
status: trusted | unverified                 # ворота проверки (наше)
receipt_ref: receipts/wiki/<id>              # ссылка на квитанцию вердикта (наше)
bound_hash: <sha256>                          # для само-инвалидации (наше)
tags: []
---
```

3. **Конвенции именования** — kebab-case для вики, дата-префикс (`2026-04-06-name.md`) для статей.
4. **Операции** — Ingest / Query / Lint (Часть III), включая нотариат и привязку.
5. **Домены интересов** — ваши темы.
6. **Профиль** — кто вы. Не вбивайте руками, пусть Claude возьмёт интервью:

> Бери у меня интервью по ОДНОМУ вопросу за раз, чтобы собрать профиль: кто я и чем занимаюсь, цели на год, как со мной общаться, сильные/слабые стороны, текущие проекты. Жди каждого ответа. В конце запиши всё в `CLAUDE.md` с заголовками.

Шаблон вики-страницы (в `wiki/_templates/`):

```markdown
# Название
## Summary
1–2 абзаца.
## Key Details
Структурированная информация — каждое фактическое утверждение с привязкой к источнику (§10).
## Related
- [[wiki/related-page]] — почему связано
## Sources
- [[raw/articles/source-file]]@<hash>
```

CLAUDE.md эволюционирует: начните с простого («super simple and flat», по Карпатому) и расширяйте.

### 9. Научите Claude диалекту Obsidian

Claude пишет на голом markdown — не знает `[[wikilinks]]`, callout-блоков, Bases, Canvas, пока не покажете. Стеф Анго (CEO Obsidian) выложил официальные Agent Skills в `kepano/obsidian-skills`. Положите содержимое репозитория в `.claude` в корне волта. Скиллы следуют открытой спецификации Agent Skills — работают и в других агентах.

## ЧАСТЬ III. РАБОЧИЙ ЦИКЛ: INGEST / QUERY / LINT

Канонический цикл Карпатого — три операции. Нотариальный слой встраивается прямо в них: **нотариат источника живёт в Ingest, проверяющий — в Lint.** Никакого отдельного «конвейера» — один цикл.

### 10. Ingest (загрузка) — с привязкой и нотариатом источника

Положили файл в `raw/`. Скажите: «Обработай новый файл в `raw/articles/2026-04-06-name.md`». Что происходит:

1. **Нотариат источника** *(наше)* — скрипт считает хеш файла, шлёт в TimeLayer, кладёт квитанцию в `receipts/raw/`. У источника появляется доказуемая «версия».
2. LLM читает источник полностью.
3. Обсуждает с вами ключевые выводы (по одному источнику за раз — так вики качественнее).
4. Создаёт/обновляет страницы в `wiki/` — **каждое фактическое утверждение несёт указатель на конкретный фрагмент источника с хешем версии** *(наше усиление дисциплины привязки)*:

```markdown
Скорость роста замедлилась во II квартале. ^[[raw/q2-report.md#L40-L48|src:q2-report@a1b2c3]]
```

5. Обновляет `wiki/index.md` и `wiki/log.md` (в лог — id квитанции).
6. Проставляет `[[ссылки]]` между связанными страницами.

Один источник может затронуть 5–15 страниц. **Вики вы руками не пишете** — это «невручную» из вашего ТЗ, а привязка к источникам — это «привязано».

**Способы загрузки в `raw/`:**

| Способ | Как | Сложность |
|---|---|---|
| Obsidian Web Clipper | кнопка в браузере → `.md` в `raw/articles/` | легко |
| Копипаст | создать `.md` вручную | легко |
| Claude Code | «сохрани эту статью в `raw/`» | легко |
| Telegram Sync | пересылка боту → `raw/telegram/` | средне |
| yt-dlp | транскрипты YouTube через CLI | средне |
| X/Twitter архив | запросить архив → распаковать | средне |

```bash
# YouTube-транскрипты
yt-dlp --write-auto-sub --sub-lang ru --skip-download -o "raw/notes/%(title)s" URL
# Резюме статьи по URL
summarize https://example.com/article > raw/articles/2026-04-06-article.md
```
Web Clipper: установите расширение, в настройках укажите волт и папку `raw/articles/`; картинки — хоткей «Download attachments» (`Ctrl+Shift+D`) скачает их в `raw/images/`.

### 11. Query (запрос) — с компаундингом в outputs/

Когда в вики 10+ страниц, спрашивайте:

- «Три главных тренда в AI-агентах на основе всего в `wiki/`?»
- «Сравни статью X и Y. Где расходятся?»
- «Брифинг на 500 слов по теме [X], только из материалов базы.»

LLM читает `wiki/index.md`, находит релевантные страницы, синтезирует ответ с цитатами. **Компаундинг:** ценные ответы сохраняйте в `outputs/` или возвращайте в `wiki/` новой страницей — каждый вопрос делает следующий ответ лучше. Важно: возвращённый в вики ответ дальше проходит Lint, как и любая страница, — иначе ошибка в ответе станет фундаментом для следующего (см. §18).

### 12. Lint (проверка) → grounding-проверяющий

У Карпатого Lint раз в месяц ищет противоречия, сироты, утверждения без источника, пропущенные ссылки, пробелы — «code review для базы знаний». **Мы превращаем Lint в полноценную проверку корректности с квитанциями.** Классические проверки Карпатого едут вместе с ней.

Lint теперь делает три вещи:
1. **Классический lint** (Карпатый): противоречия между страницами, страницы-сироты, концепции без своей страницы, битые/пропущенные ссылки, устаревшие утверждения.
2. **Grounding-проверка** *(наше)*: по каждому утверждению — следует ли оно из процитированного источника. Полные ступени — §15.
3. **Нотариат и ворота** *(наше)*: вердикт заверяется, статус `trusted` присваивается/снимается. §14, ступени 4–5.

Почему это важно: классический Lint ловит *противоречия и битые ссылки*, но НЕ ловит уверенно-неверное утверждение, которое внутренне согласовано (смысловой дрейф). Grounding ловит именно это. А git даёт откат, но не даёт ни tamper-evidence, ни авторства, ни **обязательности** проверки. Нотариальный слой закрывает ровно этот зазор.

### Просмотр в Obsidian

**Graph View** (`Ctrl/Cmd+G`): хабы (страницы с многими входящими ссылками — ключевые концепции), сироты (без связей — добавить ссылки или удалить), кластеры тем. Подсветите по `status` — увидите, где `unverified` карантин.

**Dataview** (плагин, запросы по frontmatter) — заодно панель доверия:
```dataview
TABLE type, domain, status, updated
FROM "wiki"
WHERE type != null
SORT status ASC, updated DESC
```
**Поиск:** до ~100 страниц хватает `index.md` + стандартного поиска Obsidian. Дальше — `qmd` (локальный BM25 + векторный поиск с re-ranking, есть CLI и MCP-сервер).

## ЧАСТЬ IV. НОТАРИАЛЬНЫЙ СЛОЙ КОРРЕКТНОСТИ

### 13. Заведите TimeLayer и поставьте верификатор

TimeLayer — нотариальная сеть: шлёте отпечаток (хеш), получаете маленькую (<1 КБ) квитанцию, проверяете офлайн открытым кодом. **Содержимое не уходит — только хеш.** Есть готовый продукт «Tamper-evident Agent Log» под действия ИИ-агентов.

1. Регистрация в кабинете (`cabinet.timelayer-os.com`). На старте 50 бесплатных квитанций.
2. Получите **API-токен** (держите узким и отдельным — «ключи, а не промпты»).
3. Скачайте с GitHub `timelayer-verifier` (репозиторий `TimeLayer-OS/timelayer-verifier`). Проверяет квитанцию офлайн, одной командой.
4. Точные параметры API и формат квитанции — в документации `timelayer-os.com/docs/`; детали подтвердите там или у поддержки (§19).

### 14. Пять ступеней нотариата (две из них — внутри Ingest и Lint)

**«Проверено» — не флаг, которому верят на слово, а вычисляемое свойство:** оно истинно, только когда есть подлинная квитанция, заверяющая хеш ровно от *текущего* содержимого заметки вместе с источниками и вердиктом. Подделать нельзя (квитанцию подписывает кворум независимых операторов). Пережить правку нельзя (правка меняет хеш — квитанция не сходится).

```
[1] нотариат источника   → в Ingest (§10.1): content-receipt на каждый источник
[2] письмо с привязкой    → в Ingest (§10.4): каждое утверждение с указателем + хеш версии
[3] grounding-проход      → в Lint (§15): независимый проверяющий, вердикт по утверждениям
[4] квитанция вердикта    → в Lint: заверяется СВЯЗКА (байты + версии источников + вердикт)
[5] двое ворот            → в Lint: продвижение в trusted  и  аудит само-инвалидации
```

**Ступень 4.** Только при `PASS`: `H = hash(байты_заметки ++ источники_с_версиями ++ вердикт)` → в TimeLayer → `cert`/`bundle` в `receipts/wiki/`, в frontmatter — `receipt_ref` и `bound_hash`. Это verification-receipt: привязывает *приговор* к *этим байтам*.

**Ступень 5 — двое ворот.**
- **Ворота A (продвижение):** заметка `trusted` ⇔ (а) verification-receipt проходит `timelayer-verifier`, И (б) заверённый хеш равен заново посчитанному от *текущего* содержимого + источников + вердикта. Нет — уезжает в `unverified/`. Статус **проверяется**, а не записан, поэтому подделать `status: trusted` бессмысленно — проверка (б) провалится.
- **Ворота B (само-инвалидация):** аудит-проход по каждой `trusted`-заметке заново считает хеш и сверяет. **Любая правка → отпечаток меняется → квитанция не сходится → статус слетает.** Старый PASS к новому тексту не приклеить. Это «неизменяемое» в честном смысле: не «нельзя поменять», а «поменяешь — мгновенно видно, доверие снимается само». Тот же приём ловит подмену в `raw/` (хеш версии разойдётся → grounding падает → каскад).

**(Опц.) человеческая подпись** поверх машинного grounding для критичных заметок — неотрекаемость, которой модель дать не может.

### 15. Grounding-проверяющий — полные ступени

Сердце системы. Вход: заметка `wiki/` с указателями. Выход: структурный вердикт, который заверяется на ступени 4. Сквозной принцип: **что можно проверить механически — проверяется механически, потому что код не галлюцинирует; модель — только для подлинно смысловых утверждений и только закалённая.**

**П1. Декомпозиция в атомарные утверждения.** Заметка — много утверждений. Разбейте каждое предложение на атомарные, каждое со своим указателем. «Выручка +12% во II квартале за счёт enterprise» = (1) +12% во II кв; (2) драйвер — enterprise; грунтуются отдельно. Правило: каждое предложение с фактом обязано отобразиться хотя бы в одно атомарное утверждение; lint ругается на предложения без отображения и утверждения без указателя.

**П2. Извлечение доказательства + сверка версии.** Для каждого утверждения достаньте фрагмент из `raw/` по указателю **и сверьте хеш версии**. Разошёлся → сигнал порчи (каскад из ступени 5), стоп. Ссылка не резолвится → автоматический `INSUFFICIENT`.

**П3. Маршрутизация по типу (ключевое решение).**

| Тип утверждения | Проверка |
|---|---|
| Числа, даты, количества, проценты, единицы | **Механически**: извлечь число из источника и из утверждения, сравнить программно (с допуском). Модель не спрашивать. |
| Цитаты, дословные формулировки, атрибуции | **Механически**: нормализованный подстрочный матч. |
| Существование/структура (ссылка, хеш версии) | **Механически**. |
| Логическое/смысловое следование («X вызвал Y», обобщения) | **Модель-NLI**, закалённая (П4–П5). |

Правило: можно проверить механически — *обязано* проверяться механически. Модель — запасной путь только для того, что механически не берётся.

**П4. Три исхода, не два.** SUPPORTED / **CONTRADICTED** (фрагмент противоречит — жёсткий провал) / **INSUFFICIENT** (источник молчит). Расщепление критично: «противоречит» значит «заметка врёт против источника», «недостаточно» — «нет опоры, нужен другой источник». Разные починки.

**П5. Закалка судьи (декорреляция ошибок).** Модель-судья — **самое слабое звено системы**; весь смысл механики-вперёд — опираться на него реже. Где без него никак:
1. **Другая модель, не та, что писала** (обязательно). Лучше — другое семейство: одинаковые модели дают *скоррелированную* ошибку, и квитанция её честно заверит.
2. **Самосогласованность:** k сэмплов (3–5), единогласие или жёсткий порог; разнобой → **к человеку, не авто-PASS**.
3. **Адверсариальная рамка:** просите судью искать причину, почему НЕ поддержано (стилмен отказа), а не «всё ли ок» — срезает льстивый PASS.
4. **Принудительная цитата:** судья обязан привести точный поддерживающий фрагмент. Не процитировал — не может поставить SUPPORTED. Грунтует самого судью.
5. **(сильнее) Ансамбль:** два разных семейства должны согласиться.

**П6. Схема вердикта** (машинно-разбираемый объект; целиком заверяется на ступени 4):

```json
{
  "claim": "выручка +12% во II квартале",
  "source_ref": "raw/q2-report.md#L40-L48",
  "source_version": "a1b2c3",
  "source_hash_ok": true,
  "method": "mechanical",
  "classification": "SUPPORTED",
  "evidence_span": "...рост составил 12% к I кварталу...",
  "agreement": "3/3"
}
```
Rollup заметки: **PASS** ⇔ каждое утверждение SUPPORTED И все `source_hash_ok`. Любой CONTRADICTED → **FAIL**. Любой INSUFFICIENT → **NEEDS_SOURCE**. Подделать «проверено» нельзя: изменишь вердикт — изменится хеш — квитанция не сойдётся.

**П7. Эскалации.** NEEDS_SOURCE → к Claude за лучшим источником, не нашёл → человек. CONTRADICTED → громкий флаг, карантин, строка в утренней сводке. Разнобой судьи → человек. Помечено `critical` → **человеческая подпись обязательна независимо от машинного PASS**.

**П8. Провалы самого проверяющего (честно).** Скоррелированные ошибки → механика-вперёд + кросс-семейство. Льстивый PASS → адверсариальная рамка + принудительная цитата. Декомпозиция потеряла утверждение → правило отображения предложений + lint. **Правдивость источника — вне области:** grounding проверяет «соответствует источнику», не «источник правдив». См. §18.

### 16. Как связать руками (скелет скриптов)

Готового плагина «TimeLayer для Obsidian» нет — сервис прикручивается тонким скриптом (Python ложится хорошо; стартер `coleam00/second-brain-starter` уже на markdown+Python). Ниже *скелет*: точные поля API и аргументы верификатора — из `timelayer-os.com/docs/`.

```python
import hashlib, json, subprocess, httpx, pathlib

TL = "https://api.timelayer-os.com/v1/notarize"
TOKEN = "..."  # узкий токен TimeLayer, из окружения, не в коде

def sha256_hex(b: bytes) -> str: return hashlib.sha256(b).hexdigest()
def canon(o) -> bytes: return json.dumps(o, sort_keys=True, ensure_ascii=False).encode()

def notarize(hex_digest: str) -> dict:
    r = httpx.post(TL, headers={"Authorization": f"Bearer {TOKEN}"},
                   json={"action_hex": hex_digest})       # шлём только хеш
    r.raise_for_status()
    return r.json()  # {cert_hex, bundle_hex, notarized_at} — подтвердить по докам

# Ступень 1 (в Ingest): нотариат источника
def ingest_source(path: pathlib.Path):
    d = sha256_hex(path.read_bytes())
    save_receipt(f"receipts/raw/{path.stem}", notarize(d), d)

# §15 (в Lint): grounding-проверяющий
def verify_note(note_path: pathlib.Path) -> dict:
    claims = decompose(note_path.read_text())   # П1
    out = []
    for c in claims:
        frag, src_ok = fetch_fragment(c)        # П2
        if not src_ok:
            out.append({**c, "source_hash_ok": False,
                        "classification": "INSUFFICIENT", "method": "mechanical"}); continue
        if c["kind"] in ("number","date","quote","structure"):
            res, m = check_mechanically(c, frag), "mechanical"     # П3
        else:
            res, m = check_with_judge(c, frag), "model"            # П4–П5
        out.append({**c, "source_hash_ok": True, "method": m, **res})
    return rollup(out)                            # П6

def check_with_judge(c, frag) -> dict:           # другая модель ≠ писатель; адверсариально;
    votes = [judge_once(c["text"], frag) for _ in range(5)]   # принудительная цитата; k сэмплов
    classes = {v["cls"] for v in votes}
    if classes == {"SUPPORTED"} and all(v["span"] in frag for v in votes):
        return {"classification":"SUPPORTED","evidence_span":votes[0]["span"],"agreement":"5/5"}
    if "CONTRADICTED" in classes: return {"classification":"CONTRADICTED","agreement":"split"}
    return {"classification":"INSUFFICIENT","agreement":"split"}   # → эскалация (П7)

# Ступень 4 (в Lint): квитанция вердикта (только при PASS)
def certify_note(note_path, sources, verdict):
    H = sha256_hex(note_path.read_bytes() + canon(sources) + canon(verdict))
    r = notarize(H)
    save_receipt(f"receipts/wiki/{note_path.stem}", r, H)
    set_frontmatter(note_path, receipt_ref=r["id"], bound_hash=H, status="trusted")

# Ворота B (в Lint): аудит само-инвалидации
def is_trusted(note_path, sources, verdict) -> bool:
    fm = read_frontmatter(note_path)
    if not fm.get("receipt_ref"): return False
    cert, bundle = load_receipt(fm["receipt_ref"])
    if subprocess.run(["timelayer-verifier","verify",cert,bundle]).returncode != 0:  # (а)
        return False
    H = sha256_hex(note_path.read_bytes() + canon(sources) + canon(verdict))
    return H == fm["bound_hash"]                                                      # (б)
```

`decompose`, `fetch_fragment`, `check_mechanically`, `judge_once`, `rollup` — модули под ваш формат указателей и пул моделей.

### 17. Расписание и железное правило

Когда цикл устоится — на расписание. Вкладка **Schedule**, задача на 7 утра: прогнать **Ingest** по новому в `raw/` (с нотариатом источника), затем **Lint** (классический + grounding + ворота), и написать сводку в три строки: что добавилось, что `trusted`, что упало в `unverified/`.

**Правило, которое не нарушается: доступом управляют ключи, а не промпты.** «Не удаляй это» — пожелание. Read-only ключ с узкой областью — настройка. TimeLayer — **свидетель, а не охранник**: докажет, что агент сделал, но не помешает. В строгом варианте (единственный путь агента к диску — через прокси, требующий квитанцию) он может и охранять, но только *подотчётность* и только если вы сами построите это горлышко. Живые данные — как в базовом гайде (`claude mcp add google-workspace uvx workspace-mcp --tools calendar`), тоже на ключе только-на-чтение.

## ЧАСТЬ V. РЕАЛЬНОСТЬ

### 18. Честный потолок (без него вся конструкция врёт)

- **Гарантия корректности ровно настолько хороша, насколько хорош проверяющий в воротах.** Нотариальность делает его приговор обязательным и неподделываемым — но не верным. Галлюцинирует SUPPORTED — квитанция честно зафиксирует ложный PASS. Вы не убили проблему доверия, а **перенесли** её с писателя на проверяющего. Выигрыш — только если проверяющий надёжнее: механика, где можно (П3); независимое семейство и закалка, где нельзя (П5); человек — для важного (П7). Две одинаковые модели дадут скоррелированную ошибку, и квитанция её заверит.
- **Grounding проверяет «соответствует источнику», а не «источник правдив».** Неправда в `raw/` честно протащится в `trusted`. Мусор на входе — заверенный мусор на выходе.
- **Tamper-evident ≠ tamper-proof.** Квитанции обнаруживают правку, не запрещают. Запрещает только ключ.
- **TimeLayer — молодая сеть, без формального аудита** (по их же словам, внешний security review лишь в планах). Сильная сторона: квитанции проверяются офлайн открытым кодом — лок-ина нет, старые квитанции остаются проверяемыми, даже если сервис исчезнет (новые — уже нет). Стройте так, чтобы могли уйти.
- **Карпатый сам отмечает этот потолок.** В его FAQ на «а если LLM ошибётся?» ответ — «Lint + git-откат». Lint ловит противоречия, git даёт откат, но ни tamper-evidence, ни авторства, ни обязательности проверки они не дают. Наш слой — апгрейд ровно этого места.

Что механизм **реально** даёт: превращает «доверься, тут верно» в «вот доказательство, что проверено вот этим источником, привязанное к этим самым словам, и вот ответственный», и **автоматически снимает доверие с любой незаметно изменённой заметки**. Он охраняет *процесс и цепочку ответственности*. Истину производит проверяющий — нотариальность лишь не даёт соврать, что проверка была.

### 19. Лучшие практики и контакты TimeLayer

**Делай:** загружай источники по одному и участвуй в обсуждении; ценные ответы возвращай в `wiki`/`outputs`; запускай Lint минимум раз в месяц (а с расписанием — ежедневно); ставь `[[вики-ссылки]]` везде; держи `CLAUDE.md` актуальным; вики в git (версионная история бесплатно); заверяй **только изменённые** файлы; проверяющий ≠ писатель, механика впереди модели, критичное — на человеке; ключи узкие и только-на-чтение.

**Не делай:** не редактируй `wiki` руками (это работа LLM — скажи ему); не модифицируй `raw/`; не усложняй (плоские файлы + хорошая схема > навороченный стек, «ловушка Notion»); не копи `raw/` без обработки (ценность в вики); не доверяй `trusted` как «истинно» — это «проверено названным проверяющим», см. §18; не клади токены/ключи в код.

**Контакты TimeLayer:** почта `timelayer.os@gmail.com`, Telegram `@TIMELAYEROS`, доки `timelayer-os.com/docs/`, верификатор и публичные ключи операторов — GitHub `TimeLayer-OS`. Что спросить у поддержки: (1) формат ответа `/v1/notarize` и как квитанция связывается с хешем — лежит ли digest в `bundle`, можно ли верифицировать против известного хеша (нужно для Ворот A/B); (2) поддержка произвольного **memo/metadata** в квитанции; (3) можно ли слать SHA-256 напрямую (внутри commitment на BLAKE3); (4) лимиты под ваш объём; (5) готовые примеры **gateway / fail-closed** для строгого варианта; (6) их статус по безопасности/аудиту. Что строите вы сами: дисциплину привязки, проверяющего (§15), ворота, расписание, ключи — TimeLayer даёт нотариат и верификатор, смысловую часть собираете на своей стороне.

### 20. FAQ

**Не Claude Code, а другой инструмент?** Да — Cursor, Codex, любой агент с доступом к файлам. Паттерн не привязан.

**Вики станет слишком большой для контекста?** На ~100 страниц (~400 000 слов) Карпатый говорит, что LLM справляется через `index.md` без RAG. Дальше — `qmd` для поиска.

**Чем отличается от Notion/Obsidian с плагинами?** Люди бросают вики, потому что бухгалтерия (ссылки, обновления, консистентность) занимает больше времени, чем ценность. LLM делает её бесплатно.

**Нужен ли Obsidian?** Нет, подойдёт любой редактор. Obsidian удобен графом и Dataview, но опционален — «the AI doesn't care what app you open the files in».

**Для команды?** Да: вики — git-репо, можно PR/branches, LLM обновляет из Slack/митингов. Здесь, кстати, нотариат особенно уместен: квитанции дают третьей стороне проверяемый след, кто что внёс.

**А если LLM ошибётся?** Базовый ответ Карпатого — Lint + git-откат. Наш слой добавляет к этому grounding-проверку (ловит смысловой дрейф, который Lint пропускает), tamper-evidence и авторство, и делает проверку **обязательной**, а не «раз в месяц по настроению».

**Заверять каждый файл каждый день?** Нет — только изменённые. Иначе бесплатные 50 квитанций и даже €19/5k улетят. При заверении дельты расход небольшой.

## Что у вас в итоге

Поработаете неделю — это приложение для заметок. Месяц — справочная система. Полгода — движок знаний, который **накапливается, а не переоткрывается**: каждая новая заметка связывается со всем, что уже есть. Поверх этого нотариальный слой даёт **доказуемую дисциплину**: вики, где каждое утверждение привязано к источнику, статус «проверено» нельзя нарисовать, а любая тихая правка сама себя выдаёт и снимает доверие.

Но трезво: вы получаете **охрану процесса и ответственности**, а не гарантию истины. Механизм заставляет проверку случиться, делает её неподделываемой и именной, и автоматически инвалидирует всё изменённое. Саму правильность производит проверяющий — выбор того, *кто* и *что* стоит в воротах Lint, и есть главное инженерное решение во всей сборке.

Та же подписка. Принципиально другая машина — и теперь она ещё и не даёт соврать, что проверка была.
